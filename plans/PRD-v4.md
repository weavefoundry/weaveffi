# WeaveFFI — Product Requirements Document v4 (Production Readiness)

WeaveFFI is a Rust CLI that generates multi-language FFI bindings from a single
API definition file (YAML/JSON/TOML). It targets C, C++, Swift, Android/Kotlin,
Node.js, WASM, Python, .NET, Dart, Go, and Ruby. The generated code calls into
a user-written Rust (or any C-ABI) library through a stable C ABI.

**Current state (entering v4):** The CLI has eleven working generators, ~878
tests, six samples (calculator, contacts, inventory, async-demo, events, plus
node-addon), annotated Rust extraction, incremental codegen with caching,
generator configuration via TOML, all CLI commands from PRD-v3 (`generate`,
`new`, `validate`, `extract`, `lint`, `diff`, `doctor`, `completions`,
`schema-version`), template engine with Tera, hook commands, inline generator
config, IR schema versioning at `0.2.0`, async support across all targets,
typed handles, builder pattern, listener events, iterator/streaming, builder
struct flag, deprecation/since annotations, mutable parameter annotations,
nested modules, cross-module type references, C++/Dart/Go/Ruby generators,
benchmarking infrastructure (criterion), Windows in CI, and automated
publishing to crates.io via semantic-release. The workspace is at v0.3.0 and
clippy-clean with `-D warnings` on all platforms.

Despite that breadth, several issues block a confident "production-ready"
1.0.0 release for end users:

- **Dead code in every generator.** `TypeRef::Callback` appears as a variant
  of the IR but is unreachable from any real-world IDL because
  `parse_type_ref` never constructs it. Yet ten generator crates contain ~50
  `todo!("callback ...")` panics that would crash if ever reached. This is
  rotten scaffolding from an earlier design that has been superseded by the
  module-level `callbacks:`/`listeners:` pattern and async functions.
- **`weaveffi upgrade` was checked off in PRD-v3 but never implemented.** The
  CLI has no `Upgrade` variant in its `Commands` enum.
- **Schema version drift.** `CURRENT_SCHEMA_VERSION` is `"0.2.0"` but every
  sample IDL, the `weaveffi new` template, and the README quickstart still
  use `version: "0.1.0"`.
- **Stale Cargo dependency.** `weaveffi new` writes `weaveffi-abi = "0.2"`
  into the user's `Cargo.toml` even though the workspace is at `0.3.0`.
- **CI compiles only the C and Node consumer paths.** The generated Python,
  Swift, .NET, C++, Dart, Go, Ruby, Android, and WASM bindings are never
  compiled or executed against a real cdylib in CI. They might not actually
  work end-to-end, only "look right" in `contains` assertions.
- **`merge_inline_generators` ignores most config keys.** It hardcodes a
  whitelist of nine keys and silently drops the rest (no `dart_*`, `go_*`,
  `ruby_*`, `strip_module_prefix`, `template_dir`, `pre_generate`,
  `post_generate`).
- **Examples missing for C++, Dart, Go, Ruby.** Only seven of eleven targets
  have anything in `examples/`.
- **Validation errors lack source spans.** No `miette`/`ariadne`-style
  diagnostics; users see a flat error message with no line/column.
- **Test assertions are mostly substring-based.** Almost no snapshot testing;
  silent regressions in generator output are easy to introduce.
- **No fuzzing harness** for the parser, validator, or `parse_type_ref`,
  despite this being a CLI that ingests user-supplied YAML/JSON/TOML.
- **No quality enforcement** for dependencies (`cargo deny`, `cargo audit`),
  unused deps (`cargo machete`), rustdoc warnings, or coverage in CI.
- **No `--check`, `--watch`, or `weaveffi format`** for the development
  workflow; no JSON Schema for IDE integration with the IDL.
- **`c_prefix` only partially propagates.** Function names use the prefix,
  but error/free/arena type names and the C++ generator's `extern "C"` block
  still hardcode `weaveffi_*`.
- **Incomplete governance documentation.** `CONTRIBUTING.md` exists but
  doesn't yet cover the snapshot-review or fuzz workflows. `SECURITY.md`,
  `CODE_OF_CONDUCT.md`, `ARCHITECTURE.md`, `.github/ISSUE_TEMPLATE/`, and
  `.github/PULL_REQUEST_TEMPLATE.md` are all missing. (`.ralphy/config.yaml`
  itself was refreshed out-of-band before this PRD started — do not modify
  that file during execution; it lives under `boundaries.never_touch`.)
- **No real-world non-trivial sample.** Calculator, contacts, inventory,
  events, async-demo are all toys. Real users want to see "kvstore" or "json
  parser" exercising every IDL feature in one place.
- **Doc strings often dropped.** `doc:` fields on functions/structs/fields
  are not consistently emitted as native doc comments in every target.
- **Generator output may not be deterministic.** Several generators iterate
  `HashMap`s; identical inputs may produce non-identical output across runs.

**Goal:** Take WeaveFFI from "feature-complete prototype" to "production-ready
1.0 candidate." That means: removing dead code paths, proving end-to-end
correctness in CI on every target, polishing CLI UX, locking down output
determinism with snapshot tests, hardening the parser with fuzzing, fully
documenting and shipping a real-world sample, and tightening every loose end
identified above. We are pre-1.0 — breaking changes to the IR, CLI, and
generated output are explicitly allowed and expected.

## Tasks

### Phase 1 — Eliminate `TypeRef::Callback` dead code across the workspace

The IR exposes a `TypeRef::Callback(Box<CallbackSignature>)` variant, but
`parse_type_ref` cannot construct it from any IDL string and there is no
other deserialization path that reaches it. As a result, every generator
contains `todo!("callback ...")` arms that would panic if ever hit. This
phase removes the variant entirely. The supported alternatives are:
(a) module-level `callbacks:`/`listeners:` for event/listener patterns,
(b) `async: true` functions for one-shot completion callbacks, and
(c) `iter<T>` for streaming results.

- [x] Remove `TypeRef::Callback` and `CallbackSignature` from the IR and every dependent crate. Specifically: in `crates/weaveffi-ir/src/ir.rs`, delete the `Callback(Box<CallbackSignature>)` variant from the `TypeRef` enum, delete the `CallbackSignature` struct, delete its `Hash` impl, delete the line `TypeRef::Callback(_) => "callback".to_string()` in `type_ref_to_string`, and remove any `Callback` arms in tests (search for `TypeRef::Callback` and `CallbackSignature` in `crates/weaveffi-ir/src/ir.rs`). The module-level `CallbackDef` struct (`pub struct CallbackDef { pub name: String, pub params: Vec<Param>, pub doc: Option<String> }`) and `ListenerDef` struct stay — they are reachable from `Module.callbacks` and `Module.listeners`. Update `crates/weaveffi-core/src/validate.rs` to remove any `TypeRef::Callback` arms. Update `crates/weaveffi-core/src/templates.rs` to remove the `TypeRef::Callback(_) => todo!("callback template type")` arm. Then go through every generator crate and remove the `TypeRef::Callback(_) => todo!(...)` arms entirely (since the variant no longer exists, they are no longer needed and will be compile errors): `crates/weaveffi-gen-c/src/lib.rs` (3 arms), `crates/weaveffi-gen-cpp/src/lib.rs` (8 arms), `crates/weaveffi-gen-swift/src/lib.rs` (2 arms), `crates/weaveffi-gen-android/src/lib.rs` (13 arms), `crates/weaveffi-gen-node/src/lib.rs` (5 arms), `crates/weaveffi-gen-wasm/src/lib.rs` (5 arms), `crates/weaveffi-gen-python/src/lib.rs` (7 arms), `crates/weaveffi-gen-dotnet/src/lib.rs` (3 arms), `crates/weaveffi-gen-dart/src/lib.rs` (any arms), `crates/weaveffi-gen-go/src/lib.rs` (any arms), `crates/weaveffi-gen-ruby/src/lib.rs` (any arms), `crates/weaveffi-cli/src/scaffold.rs` (3 arms). For each generator, also remove the now-unused `Box<CallbackSignature>` imports and any helper functions that only existed to generate placeholder callback code. After removing, `cargo build --workspace` must succeed; if any `match` becomes non-exhaustive, fix the patterns to cover only the remaining variants. Add a regression test in `crates/weaveffi-ir/src/ir.rs` named `parse_type_ref_does_not_yield_callback` that asserts `parse_type_ref("callback")` returns `Ok(TypeRef::Struct("callback".into()))` (i.e., a user-defined struct type named `callback`, not a built-in). Also add a documentation paragraph to `crates/weaveffi-ir/src/ir.rs` above the `TypeRef` enum explaining that callback-style behavior is expressed via module-level `callbacks:`/`listeners:` and `async: true` functions, not via a callback type. Update `docs/src/reference/idl.md` to remove any mention of a `callback` type as a `TypeRef`. Run `cargo test --workspace` to verify nothing is broken. Run `rg "todo\!" crates/` after the change and confirm only legitimate `todo!()` macros remain (e.g., in scaffold output and the `weaveffi new`/extract test fixtures, plus the user-facing documentation in `cmd_new`'s printed instructions). Specifically, every `todo!("callback ...")` must be gone.

### Phase 4 — Source-span-aware diagnostics with `miette`

Today, parse and validation errors are flat strings with no line/column
context. Users get `"unknown type ref: Foo"` with no indication of which
of their 200 lines of YAML is at fault. Adopt `miette` for rich CLI
diagnostics with snippets and carets.

- [x] Add `miette` to the workspace and produce span-aware error output. Add `miette = { version = "7", features = ["fancy"] }` to `[workspace.dependencies]` in the root `Cargo.toml`. Add it as a dependency of `weaveffi-ir`, `weaveffi-core`, and `weaveffi-cli`. In `crates/weaveffi-ir/src/parse.rs`, change `ParseError` to a `#[derive(Diagnostic, thiserror::Error)]` enum where each variant carries `#[source_code]` (the input string, owned via `String`) and `#[label]` (a `SourceSpan`). For YAML parsing, use `serde_yaml::Error::location()` to extract `(line, col)` and convert to a byte offset into the source — write a helper `fn line_col_to_offset(src: &str, line: usize, col: usize) -> usize`. For JSON, use `serde_json::Error::line()`/`column()`. For TOML, use `toml::de::Error::span()`. Wrap each parser variant accordingly. In `crates/weaveffi-core/src/validate.rs`, add `#[derive(Diagnostic)]` to `ValidationError` and provide `#[help]` strings. Validation errors don't currently carry spans, so pass `&str` source through to the validator: change `validate_api(api: &mut Api)` to `validate_api(api: &mut Api, source: Option<(&str, &str)>)` (filename + contents) and propagate a best-effort byte range when the error references a known IR field. To reach this, augment the IR types with optional spans during parsing — but that is an invasive refactor; instead, re-search the source string for the offending name (e.g., `format!("\"{}\"", offending_name)` for YAML/JSON, with a fallback) to compute a span for the `#[label]`. Where a span cannot be computed, omit the label (still produces nicer output). In `crates/weaveffi-cli/src/main.rs`, replace the bespoke `format_parse_error` and `format_validation_error` with `miette::Report::new(err)`-style returns from `cmd_*`. Use `miette::set_hook` in `main` to install the fancy printer. Verify that running `weaveffi validate <bad-file>` produces colored output with a snippet and caret on the offending line in a terminal (best effort — there is no automated way to test ANSI in CI; instead, write a non-color test that asserts the rendered string contains the offending identifier and the line number). Add tests in `crates/weaveffi-cli/tests/cli_diagnostics.rs`: `parse_yaml_error_includes_line` runs the binary on a YAML with a syntax error, captures stderr, and asserts the line number appears; `validate_unknown_typeref_includes_offending_name` similarly. Run `cargo test --workspace` to verify nothing is broken.

fix.

Add a workspace-level test `crates/weaveffi-cli/tests/determinism.rs::generator_output_is_byte_identical_across_runs` that runs each generator on the kitchen-sink fixture twice into two temp dirs and asserts that every output file is byte-identical. ### Phase 19 — Async robustness: GC pinning, cancellation, leak tests

Async generation is implemented but never load-tested. Audit the
.NET/Swift/Node/Python/Kotlin/C++/Dart/Go/Ruby/WASM async paths for
common pitfalls (callback dropped before resolved, GC reclaims pinned
delegate, double-resume, leak on cancellation).

- [x] Audit and stress-test async correctness across every async-capable target. In each generator's async wrapper code: (1) Ensure the user-provided context pointer's owner is exactly one entity that frees it on the callback path AND on the cancellation path, never both. (2) For .NET, ensure `GCHandle.Alloc(delegate, GCHandleType.Normal)` is matched by `GCHandle.Free(handle)` in the C callback after resolving the TaskCompletionSource. For Swift, ensure `Unmanaged<CheckedContinuation>.passRetained(...)` is balanced by `Unmanaged.fromOpaque(ptr).takeRetainedValue()` in the callback. For Node, ensure `napi_create_reference` for the deferred is matched by `napi_delete_reference` after `napi_resolve_deferred`/`napi_reject_deferred`. For Python, ensure `Py_INCREF` on the future is balanced by `Py_DECREF` after `set_result`/`set_exception`, and that the call goes through `PyGILState_Ensure`/`PyGILState_Release` since the C callback may run on a non-GIL thread. For Kotlin, ensure JNI `NewGlobalRef` on the continuation is matched by `DeleteGlobalRef`. For C++, ensure the captured `std::promise` is moved into the callback closure and the closure heap allocation is freed exactly once. For Dart, ensure `NativeCallable.listener` is closed in a `finally`. For Go, ensure cgo callback lifetimes don't outlive the channel. For Ruby, ensure `rb_global_variable` registration is removed when the FFI callback completes. For WASM, ensure `wasm.addFunction` table entries are removed via `wasm.removeFunction` after the promise resolves.
