# WeaveFFI — Product Requirements Document v3

WeaveFFI is a Rust CLI that generates multi-language FFI bindings from a single
API definition file (YAML/JSON/TOML). It targets C, Swift, Android/Kotlin,
Node.js, WASM, Python, and .NET. The generated code calls into a user-written
Rust library through a stable C ABI.

**Current state:** The CLI has seven working generators, ~495 tests, three
samples (calculator, contacts, inventory), annotated Rust extraction, incremental
codegen with caching, generator configuration via TOML, CLI commands (generate,
new, validate, extract, lint, diff, doctor), and automated publishing to
crates.io via semantic-release. The WASM generator does not handle the `out_err`
error parameter in its JS wrappers. The Node N-API addon has placeholder stub
bodies (`// TODO: implement`). The `GeneratorConfig` fields `wasm_module_name`,
`c_prefix`, and `strip_module_prefix` are defined but not wired into their
respective generators. There is no shell completions command, no typed handles,
no benchmarking infrastructure, no C++/Dart/Go/Ruby targets, no async support,
no template engine, and no plugin/hook system.

**Goal:** Implement all future roadmap goals: quality polish, safety and
performance, C++ target, template engine and extensibility, async and callback
support, Dart/Flutter target, Go target, Ruby target, advanced IR patterns
(streaming, callbacks, builders, cross-module types), and stability milestone
(Windows CI, security audit, migration tooling, per-target tutorials).

### Phase 1 — WASM generator: error handling alignment

The WASM JS wrappers currently call C ABI functions without passing or checking
`weaveffi_error`. Every other generator handles errors; the WASM generator must
do the same.

- [x] Add error handling to the WASM JS function wrappers in `crates/weaveffi-gen-wasm/src/lib.rs`. In `render_wasm_js_stub`, add two helper functions to the generated JS: (1) `_allocError(wasm)` that calls `wasm._raw.weaveffi_alloc(8)` to allocate an 8-byte error struct in linear memory (4 bytes for i32 code + 4 bytes for i32 message pointer) and returns the pointer, (2) `_checkError(wasm, errPtr)` that reads the i32 at `errPtr` from `wasm._raw.memory.buffer` via `new Int32Array(buffer, errPtr, 1)[0]`; if non-zero, reads the message pointer at `errPtr + 4` via `new Uint32Array(buffer, errPtr + 4, 1)[0]`, decodes it with `_decodeString`, calls `wasm._raw.weaveffi_error_clear(errPtr)`, and throws `new Error(\`WeaveFFI error \${code}: \${msg}\`)`. In `emit_js_function_wrapper`, update every generated wrapper to: (a) call `const _err = _allocError(wasm)`, (b) pass `_err` as the last argument to the raw WASM export, (c) call `_checkError(wasm, _err)` after the call. Also emit a `weaveffi_alloc` and `weaveffi_error_clear` import expectation in the README documentation. Add tests: `wasm_js_has_error_helpers` asserts `_allocError` and `_checkError` appear in the generated JS, `wasm_js_function_passes_err` asserts a generated function wrapper contains `_err` and `_checkError`. Run `cargo test --workspace` to verify nothing is broken.

- [x] Update the WASM `.d.ts` to document that functions may throw. In `crates/weaveffi-gen-wasm/src/lib.rs`, in `render_wasm_dts`, add a JSDoc comment `/** @throws {Error} if the native call fails */` above each method in the `WeaveFFIModule` interface. ### Phase 3 — Node N-API addon: functional implementation

The Node generator writes a `weaveffi_addon.c` with `// TODO: implement` stubs
for every function. These must become functional N-API wrappers that extract
arguments, call the C ABI function, check errors, and return results.

- [x] Implement functional N-API bodies in the Node generator. In `crates/weaveffi-gen-node/src/lib.rs`, rewrite `render_addon_c` so that each generated `static napi_value Napi_weaveffi_{module}_{function}(napi_env env, napi_callback_info info)` function body contains working N-API code. For each function: (1) declare `size_t argc = N` and `napi_value args[N]` where N is the param count, call `napi_get_cb_info(env, info, &argc, args, NULL, NULL)`, (2) for each param, extract the value using the appropriate N-API getter — `I32`/`U32`/`I64` → `napi_get_value_int32`/`napi_get_value_uint32`/`napi_get_value_int64`, `F64` → `napi_get_value_double`, `Bool` → `napi_get_value_bool`, `StringUtf8` → `napi_get_value_string_utf8` (first call with NULL to get length, then allocate and call again), `Enum(name)` → `napi_get_value_int32`, `Handle` → `napi_get_value_int64`, (3) declare `weaveffi_error err = {0}`, (4) call the C ABI function `weaveffi_{module}_{function}(...)` passing `&err` as the last argument, (5) check `err.code != 0` — if so, call `napi_throw_error(env, NULL, err.message)` then `weaveffi_error_clear(&err)` and `return NULL`, (6) convert the return value to a `napi_value` — `I32` → `napi_create_int32`, `Bool` → `napi_get_boolean`, `StringUtf8` → `napi_create_string_utf8` then free the C string with `weaveffi_free_string`, `Handle` → `napi_create_int64`, `Struct` → `napi_create_int64` (opaque handle as int64), (7) return the `napi_value`. For functions with no return (`returns` is `None`), return `undefined` via `napi_get_undefined`. For `List` params and returns, use a loop with `napi_get_array_length` / `napi_get_element` or `napi_create_array` / `napi_set_element`. For `Optional` params, check `napi_typeof` for `napi_null` or `napi_undefined`. Add tests: `node_addon_has_no_todo` asserts the generated `weaveffi_addon.c` does NOT contain `// TODO: implement`, `node_addon_extracts_args` asserts the generated code contains `napi_get_cb_info`, `node_addon_checks_error` asserts it contains `err.code`.
