# WeaveFFI — Product Requirements Document v2 (0.2.0 – 0.5.0)

WeaveFFI is a Rust CLI that generates multi-language FFI bindings from a single
API definition file (YAML/JSON/TOML). It targets C, Swift, Android/Kotlin,
Node.js, and WASM. The generated code calls into a user-written Rust library
through a stable C ABI.

**Current state:** The CLI supports flat functions *and* complex types (structs,
enums, optionals, lists). It has generators for all five targets, two samples
(calculator and contacts), over 200 tests, documentation via mdbook, and a
semantic-release pipeline. The IR, validation, and code generation for the
MVP type system are functionally complete. However, several generators have
edge-case gaps, the WASM generator is static documentation only (not
API-driven), there is no support for reading annotated Rust source as input,
no incremental codegen, no Python or .NET targets, and the project has never
been published to crates.io.

**Goal:** Implement the remaining 0.2.0 through 0.5.0 roadmap goals: finish
generator completeness and packaging (0.2.0), add annotated Rust extraction
and template configuration (0.3.0), add incremental codegen and DX polish
(0.4.0), and add Python and .NET generators, inline helpers, and release
automation (0.5.0).

## Tasks

### Phase 1 — Resolve TypeRef::Enum from IDL strings

Currently `parse_type_ref` maps all unknown identifiers to `TypeRef::Struct(name)`.
This means enum references like `type: ContactType` parse as `Struct("ContactType")`
rather than `Enum("ContactType")`. The validator treats both the same, but
generators must guess. Fix this by adding a resolution pass.

- [ ] Add a `resolve_type_refs` function to `crates/weaveffi-core/src/validate.rs` that walks every `TypeRef` in every function param, return type, and struct field, and if it encounters `TypeRef::Struct(name)` where `name` matches an enum defined in the same module, replaces it with `TypeRef::Enum(name)`. This must handle nested types: `Optional(Struct("Color"))` should become `Optional(Enum("Color"))` if `Color` is an enum. The function signature should be `pub fn resolve_type_refs(api: &mut Api)`. Call it from `validate_api` *after* all validation passes (so the input is known-valid) — change `validate_api` to take `&mut Api` instead of `&Api`. Update all call sites in `crates/weaveffi-cli/src/main.rs` to pass `&mut api` instead of `&api`. Add tests in `crates/weaveffi-core/src/validate.rs`: (1) `resolve_enum_ref_in_function_param` — a function param with `Struct("Color")` where `Color` is an enum gets resolved to `Enum("Color")`, (2) `resolve_enum_ref_in_optional` — `Optional(Struct("Color"))` becomes `Optional(Enum("Color"))`, (3) `struct_ref_not_changed` — `Struct("Contact")` where `Contact` is a struct stays as `Struct("Contact")`, (4) `resolve_enum_ref_in_struct_field` — a struct field with `Struct("Color")` where `Color` is an enum gets resolved. Run `cargo test --workspace` to verify nothing is broken.
